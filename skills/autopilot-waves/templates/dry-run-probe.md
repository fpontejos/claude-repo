# Dry-Run Probe Template

**When to use:** before dispatching a wave whose acceptance criteria include numerical oracles (computed thicknesses, geometric chains, parsing-derived constants, formula outputs) that the **plan does not pin**. The probe writes a throwaway script that simulates the proposed algorithm against real production data, emits a per-fixture oracle table, and that table becomes the source for `pytest.approx(...)` assertions in test-writing agents.

**Origin:** introduced during a data-converter run whose wave spec left multiple numerical ambiguities unresolved (an inner-vs-outer column split, derived-value chaining, a load-bearing category promotion). The probe caught a real correctness bug *before* any agent dispatch — the draft would have collapsed two categories into the same column.

---

## When NOT to use

- Wave is pure structural (new module with a clear API contract; no fixture-driven oracle).
- Plan already pins every concrete number the tests will assert.
- Wave has a stable cross-boundary test as gate AND the math is trivial enough to spec inline.
- Time pressure / single-shot probe would take longer than two agent retries (>30 min).

Use the probe when **the plan would force agents to guess constants**.

---

## Workflow (orchestrator)

1. **Identify the open numerical questions.** Read the plan and the wave's test acceptance criteria. List every place an agent would need to *decide* a constant or expected value (e.g., "what does the chain produce for fixture X?", "what's the right parsing fallback?", "which column gets read for category Y?").
2. **Write the probe script** at `scripts/_scratch/<wave>_probe.py` (or analogous throwaway dir for the project). Reuse production code paths (`load_recipe_spine`, `classify_entry`, packing/parsing helpers shipped by prior waves). Do not reimplement — that defeats the point of grounding in reality.
3. **Run against ALL pinned fixtures**, not just one. Cross-fixture comparison surfaces edge cases (e.g., "this rule works for 4 of 5 fixtures, the 5th breaks because…").
4. **Dump output to `docs/plans/<plan-dir>/<wave>-dry-run.md`** as a markdown table per fixture. Each row = one chain step / one expected value, with both inputs and the computed result. Include warnings the production code would emit.
5. **Update the wave's design doc** (`<wave>-analysis.md` or equivalent) with the resolved findings — what was ambiguous, what the probe revealed, what rule applies. Move open questions to "resolved by dry-run" subsection.
6. **Update agent dispatch prompts** (`<wave>-prompts.md` or the dispatch text) to:
   - Replace hand-wavy "agent decides" instructions with concrete `pytest.approx(...)` values
   - Add a cross-check step in the implementation agent's verification: "re-run the probe, compare to orchestrator's pinned output"
   - Cite the dry-run.md file path so agents can read the chain
7. **Iterate the probe** if the first run surfaces a new ambiguity. Probes are cheap; iterate until the rules are unambiguous before dispatching agents.

    return {
        "id": fixture_id,
        "label": label,
        "inputs": {...},

        "outputs": {...},
        "warnings": [...],

## Oracle dump format

The dry-run.md file's job is to be **readable as a reference document** by both humans and the test-writing agent. One section per fixture; one table per chain or rule. Bold the final computed values so agents can scan-match.

```markdown
# <Wave> dry-run results

Generated by `scripts/_scratch/<wave>_probe.py`. Throwaway.

## FIXTURE_NAME_1 (id=12345)

- relevant_input_1: value
- relevant_input_2: value
- **bundle_diameter / intermediate_constant: 2.7584**

### Chain

| step | category | input | prev | computed | warning |
|---|---|---|---|---|---|
| 0 | layer_type_A | 4.0 | 2.76 | **0.62** |  |
| 1 | layer_type_B | 5.5 | 4.0 | **0.75** |  |
| 2 | layer_type_C | None | 5.5 | **placeholder** | THICKNESS_INVALID |
```

Wrap-up summary table cross-fixture is also useful:

```markdown
## Per-fixture oracle summary

| Fixture | bundle | layer_1 | layer_2 | last_layer | warnings |
|---|---|---|---|---|---|
| F1 | 2.04 | 0.06 | 0.37 | (=last) | none |
| F2 | 4.41 | 0.50 | 0.50 | (=last) | none |
| F3 | 2.56 | placeholder + warn | 0.30 | (=last) | 1 THICKNESS_INVALID |
```

## Wiring into agent prompts

### Implementation agent (e.g., D.1)

Add a verification step:

```markdown
**Dry-run cross-check:** the orchestrator built a per-fixture oracle in
`docs/plans/<plan-dir>/<wave>-dry-run.md` using `scripts/_scratch/<wave>_probe.py`.
After your changes, the production code path should produce the same per-step
values (within tolerance). Re-run the probe:

    uv run python scripts/_scratch/<wave>_probe.py

Compare to the pinned version. If they disagree, cite specific row mismatches
in your return summary — your implementation diverged from the spec.
```

### Test-writing agent (e.g., D.2)

Replace "agent decides expected values" with pinned numbers from the dry-run:

```markdown
**Oracle values pinned by dry-run probe** (see docs/plans/<plan-dir>/<wave>-dry-run.md
for the full chain per fixture):

- For FIXTURE_1: `result.layer[0].value == pytest.approx(0.0619, abs=0.005)`
- For FIXTURE_2: `result.layer[-1].value == pytest.approx(0.4975, abs=0.005)`
- ...

Read the dry-run.md file before writing the assertions. If your computed expected
diverges from the dry-run, STOP and report — likely the rule changed between
probe and dispatch.
```

---

## Cleanup

- `scripts/_scratch/` is **gitignored** (or should be — add to `.gitignore` if not). The probe stays there during the wave; remove after the wave ships and tests are green.
- `docs/plans/<plan-dir>/<wave>-dry-run.md` lives with the plan; keep it. Future Wave G / Phase H may want to re-run with updated inputs.
- The probe is a frozen-in-time snapshot — note in its header which BOM/DB/code version it was run against. If the underlying data shifts, the probe results stale.

---

## Anti-patterns

| Don't | Do |
|-------|----|
| Reimplement the algorithm in the probe | Reuse production helpers — the probe is testing the rule, not the implementation |
| Run on one fixture | All pinned fixtures. Cross-fixture variance is where ambiguities hide |
| Skip the dump file; "I'll just remember the numbers" | Write the markdown. Agents read files, not chat history |
| Dispatch the implementation agent before re-running the probe | Iterate the probe until the rule is unambiguous |
| Use the probe as the test oracle directly | Probe values feed into pytest assertions; they are not themselves the gate |
| Keep the probe forever | Remove after the wave ships; the dry-run.md stays |

---

## Cost / benefit

- **Cost:** ~15–30 min orchestrator time per wave.
- **Avoided:** at least one agent retry (15–20 min) per ambiguous oracle, plus the cost of catching the bug later in a gate failure or — worse — after commit.
- **Caught in practice:** one wave's dry-run caught an inner-vs-outer column-split bug in a data mapping that would have produced silently-wrong values for ~30% of records. Without the probe, the wave would have shipped mapping every entry to the outer column; the gate would have passed (the pinned fixtures didn't cover the split) and the bug would have surfaced months later in downstream comparisons.

Use the probe whenever the wave's correctness depends on values the plan doesn't pin.
